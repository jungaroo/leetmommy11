Setting State Using State

Weâve established that setState() is asynchronousâ¦

So: itâs risky to assume previous call has finished when you call it. Also, React will sometimes batch (squash together) calls to setState together into one for performance reasons.

Therefore, if a call to setState() depends on current state,
the safest thing is to use the alternate âcallback formâ.

setState Callback Form

this.setState(callback)

Instead of passing an object, pass it a callback with the current state as a parameter.

The callback should return an object representing the new state.

this.setState(curState => ({ count: curState.count + 1 }));

Abstracting State Updates

The fact that you can pass a function to this.setState lends itself nicely to a more advanced pattern called functional setState.

Basically you can describe your state updates abstractly as separate functions. But why would you do this?

// elsewhere in the code
function incrementCounter(prevState) {
  return { count: prevState.count + 1 };
}
// somewhere in the component
this.setState(incrementCounter);

Because testing your state changes is as simple as testing a plain function:

expect(incrementCounter({ count: 0 })).toEqual({ count: 1 });

We will revisit this type of pattern when we talk about Redux.

A Simple Counter Component

demo/simple-counter/src/SimpleCounter/SimpleCounter.js
class SimpleCounter extends Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
    this.clickUp = this.clickUp.bind(this);
    this.clickDown = this.clickDown.bind(this);
  }

  clickUp(event) { this.setState(countUp) }

  clickDown(event) { this.setState(countDown) }

  render() {
    return (
      <div>
        <h1>Count: {this.state.count}</h1>
        <button id="count-up" onClick={this.clickUp}>
          Count Up
        </button>
        <button id="count-down" onClick={this.clickDown}>
          Count Down
        </button>
      </div>
    );
  }
} // end

Enzyme .state() and .props()

Enzyme can peer into a componentâs state or props.

With the component shallow-mounted in wrapper, just call the .state() or .props() methods to get the objects.

Testing State Changes

The format is generally:

  • assert the default state
  • simulate an event that causes a state change
  • assert the changed state
demo/simple-counter/src/SimpleCounter/SimpleCounter.test.js
    // assert we have a default state
    expect(wrapper.state().count).toBe(0);
    // click on the up button
    upButton.simulate('click');
    // assert the state has changed
    expect(wrapper.state().count).toBe(1);

Full Example

demo/simple-counter/src/SimpleCounter/SimpleCounter.test.js
  // count up
  it('updates state when counting up', function() {
    const wrapper = shallow(<SimpleCounter />);
    // find the count-up button
    const upButton = wrapper.find('#count-up');
    // assert we have a default state
    expect(wrapper.state().count).toBe(0);
    // click on the up button
    upButton.simulate('click');
    // assert the state has changed
    expect(wrapper.state().count).toBe(1);
    // done!
  });

Bonus: Testing Functional setState

We also abstracted our state updates to pure functions: countUp and countDown

demo/simple-counter/src/SimpleCounter/SimpleCounter.js
// abstracted state updates
function countUp(state) {
  return { count: state.count + 1 };
}

function countDown(state) {
  return { count: state.count - 1 };
}

This makes unit testing super easy! No selecting/simulating required!

demo/simple-counter/src/SimpleCounter/SimpleCounter.test.js
  /* unit tests for pure state-updating functions */

  describe('countUp', function() {
    it('counts from 0 to 1', function() {
      expect(countUp({ count: 0 })).toEqual({ count: 1 });
    });
  });

  describe('countDown', function() {
    it('counts from 1 to 0', function() {
      expect(countDown({ count: 1 })).toEqual({ count: 0 });
    });
  });

Mutable Data Structures in State

Mutable Data Structures

Up til now, we have just been setting state to primitives: numbers and strings mainly.

But component state also commonly has keys with objects, arrays, and arrays of objects.

this.state = {
  // store an array of todo objects
  todos: [
    { task: 'do the dishes', done: false, id: 1 },
    { task: 'vacuum the floor', done: true, id: 2 }
  ]
};

You have to be extra careful modifying your array of objects!

completeTodo(id) {
  const theTodo = this.state.todos.find(t => t.id === id);
  theTodo.done = true; // NOOOOO

  this.setState({
    todos: this.state.todos // bad
  });
}

If you mutate any of your nested data structures it will mess up React.

Immutable State Updates

A much better way is to make a new copy of the data structure in question. We can use any pure function to do thisâ¦

completeTodo(id) {

  // Array.prototype.map returns a new array
  const newTodos = this.state.todos.map(todo => {
    if (todo.id === id) {
      // make a copy of the todo object with done -> true
      return { ...todo, done: true };
    }
    return todo;  // old todos can pass through
  });

  this.setState({
    todos: newTodos // setState to the new array
  });
}

Pure functions such as .map, .filter, and .reduce are your friends. So is the â¦spread operator.

There is a slight efficiency cost due to the O(N) space/time required to make a copy, but itâs almost always worth it to ensure that your app doesnât have extremely difficult to detect bugs due to mischevious side effects.

Immutable State Summary

  • While it sounds like an oxymoron, immutable state just means that there is an old state object and a new state object that are both snapshots in time.
  • The safest way to update state is to make a copy of it, and then call this.setState with the new copy.
  • This pattern is a good habit to get into for React apps and required for using Redux.

Designing State

Being able to design the state of a React application (or any modern web app) is a challenging skill that takes lots of practice.

However, there are some easy best-practices that we can talk about in this section to give you a jump-start.

State Should Be Minimized

In React, you want to try to put as little data in state as possible.

Litmus test

  • does x change? If not, x should not be part of state. It should be a prop.
  • is x already captured by some other value y in state or props? Derive it from there instead.

Letâs pretend weâre modelling a Personâ¦

  • Does Mattâs birthday ever change? How is that even possible!
  • Mattâs age does change, however if we had this.props.birthday we could easily derive it from that.
  • Therefore, the only property here that is truly stateful is arguably mood (although Matt might dispute this ð).

State Should Live On the Parent

As previously mentioned, we want to support the âdownward data flowâ philosophy of React.

In general, it makes more sense for a parent component to manage state and have a bunch of âdumbâ stateless child display components.

This makes it easier for debugging purposes, because the state is centralized, and it becomes more predictable where to find state:

Is the current component stateless? Find out what is rendering it. Thereâs the state.

Todo Example:

class TodoList extends Component {
  constructor(props) {
    super(props);
    this.state = {
      todos: [
        { task: 'do the dishes', done: false, id: 1 },
        { task: 'vacuum the floor', done: true, id: 2 }
      ]
    };
  }
  /* ... lots of other methods ... */
  render() {
    return (
      <ul>
        {this.state.todos.map(t => <Todo {...t} />)}
      </ul>
    );
  }
}

TodoList is a smart parent with lots of methods, while the individual Todo items are just <li> tags with some text and styling.
